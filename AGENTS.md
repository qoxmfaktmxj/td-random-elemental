# AGENTS.md

## 0. 목적

이 문서는 이 프로젝트에서 작업하는 AI Agent(Codex, Claude Code 등)가
일관된 방식으로 개발하도록 강제하기 위한 규칙 문서다.

이 프로젝트는 단순 기능 구현이 아니라
"3D 캐릭터 기반 랜덤 합성 엘리멘탈 타워디펜스 게임"을 만드는 것이 목적이다.

특히 다음을 매우 중요하게 여긴다:

- 그래픽 퀄리티 (시각적 만족감)
- 명확한 구조 설계
- 작은 단위로 안전하게 개발
- 과도한 추측 금지

---

## 1. 핵심 개발 원칙

### 1.1 절대 추측하지 말 것
- 기존 코드, 파일, 구조를 반드시 확인한 후 작업한다.
- 없는 기능을 있다고 가정하지 않는다.
- 애매하면 최소한의 구조부터 만든다.

---

### 1.2 작은 단위로 작업
- 한 번에 큰 기능을 만들지 않는다.
- 항상 "하나의 책임"만 가지는 코드 단위로 나눈다.
- 기능 추가 시 기존 코드 영향 범위를 최소화한다.

---

### 1.3 MVP 우선
- 항상 MVP 기준으로 먼저 구현한다.
- 불필요한 확장, 미래 기능은 구현하지 않는다.
- "지금 당장 필요한 것"만 만든다.

---

### 1.4 그래픽/연출 우선 고려
이 프로젝트는 다음을 중요하게 본다:

- "보는 맛"
- "강해졌다는 체감"
- "이펙트 차이"

따라서:
- 단순 수치 증가만 구현하지 말 것
- 가능하면 시각적 차이를 함께 고려할 것

---

## 2. 기술 스택 기준

- Engine: Unity (C#)
- 3D 기반
- ScriptableObject 활용 가능
- 모듈형 구조 지향

---

## 3. 프로젝트 구조 원칙

### 3.1 기능별 분리

코드는 다음 기준으로 분리한다:

- Character (플레이어)
- Tower (타워)
- Enemy (적)
- Wave (웨이브)
- Element (속성)
- UI
- Core (공통 시스템)

---

### 3.2 의존성 최소화

- 각 시스템은 최대한 독립적으로 구성한다.
- Tower가 Enemy 내부 구현을 직접 알지 않도록 한다.
- Element는 "효과 데이터"로만 사용한다.

---

### 3.3 데이터와 로직 분리

가능하면 다음 구조를 따른다:

- 데이터: ScriptableObject
- 로직: MonoBehaviour / Service

---

## 4. 작업 방식 규칙

### 4.1 작업 전 반드시 할 것

- 현재 프로젝트 구조 확인
- 관련 코드 파일 확인
- 기존 패턴 확인

---

### 4.2 작업 단위 규칙

각 작업은 반드시 다음 형태로 진행한다:

1. 목적 정의
2. 필요한 파일 목록
3. 코드 작성
4. 테스트 방법 제시
5. 변경 요약

---

### 4.3 한 번에 하나의 Task만 수행

- 여러 기능을 동시에 구현하지 않는다.
- 항상 단일 Task 기준으로 작업한다.

---

## 5. 코드 작성 규칙

### 5.1 명확한 네이밍

- 의미 없는 이름 금지 (temp, test, data 등)
- 역할 기반 네이밍 사용

예:
- TowerAttackController
- EnemyMovementController
- ElementEffectData

---

### 5.2 책임 분리

하나의 클래스는 하나의 역할만 가진다.

❌ 잘못된 예:
- TowerManager가 공격, UI, 데이터까지 다 처리

✅ 올바른 예:
- TowerAttack
- TowerData
- TowerUI

---

### 5.3 하드코딩 금지

- 수치는 코드에 직접 박지 않는다.
- 가능하면 데이터로 분리한다.

---

## 6. 게임 시스템 설계 규칙

### 6.1 타워 구조

- 타워는 "역할" 기반으로 구성한다.
- 속성(Element)은 "추가 효과"로만 사용한다.

예:
- Attack Tower + Fire Element → 화상 효과
- Slow Tower + Water Element → 둔화 강화

---

### 6.2 합성 시스템

- 동일 타워 3개 → 상위 단계 1개
- 단계는 1~5까지 존재

중요:
- 단계 상승 시 반드시 "시각적 변화"를 고려한다.

---

### 6.3 랜덤 시스템

- 타워는 랜덤으로 생성된다.
- 단, 초반 망 방지를 위한 보정이 필요하다.

---

### 6.4 캐릭터

- 플레이어는 캐릭터를 직접 조작한다.
- 캐릭터는 보조 역할이다.
- 핵심은 타워 시스템이다.

---

## 7. 그래픽 / 연출 가이드

### 7.1 단계별 성장 체감

각 타워는:

- 크기 증가
- 이펙트 증가
- 색상 강화
- 오라 추가

가 반드시 있어야 한다.

---

### 7.2 속성별 시각 구분

- Fire: 붉은색, 불꽃, 폭발
- Water: 파란색, 흐름, 얼음
- Earth: 갈색, 돌, 충격파
- Dark: 보라/검정, 연기, 왜곡

---

### 7.3 합성 연출

합성 시:

- 파티클 효과
- 빛 효과
- 크기 변화
- 짧은 연출

을 반드시 고려한다.

---

## 8. 테스트 및 검증

코드 작성 후 반드시:

- 실행 가능 여부 확인
- NullReference 방지
- 기본 동작 테스트

가능하면:
- Play Mode 테스트 기준 제시

---

## 9. 금지 사항

- 전체 시스템을 한 번에 구현
- 기존 코드 무시하고 새로 작성
- 불필요한 최적화
- 사용되지 않는 코드 작성
- 과도한 추상화

---

## 10. 출력 형식 규칙

항상 아래 형식으로 답변한다:

1. 작업 목적
2. 변경/추가 파일
3. 코드
4. 테스트 방법
5. 변경 요약
6. 남은 리스크

---

## 11. 최종 목표

이 프로젝트의 목표는:

- 단순한 기능 구현이 아닌
- "보는 맛 + 성장 체감 + 반복 플레이"가 가능한
- 완성도 있는 3D 타워디펜스 게임을 만드는 것이다.

AI는 항상 이 목표를 기준으로 판단하고 작업해야 한다.
