## 1. 개발 착수용 MVP 잠금 범위

### 1.1 이번에 실제로 만들 MVP

이 문서 기준 MVP는 "게임의 핵심 재미가 증명되는 가장 작은 완성본"이다. 이전 설계 문서보다 한 번 더 줄여서 아래 범위로 잠근다.

| 영역 | MVP 잠금안 |
|---|---|
| 플랫폼 | PC 싱글플레이 |
| 전투 공간 | 1개 맵, 1개 경로, 1개 코어 |
| 카메라 | 세미 아이소메트릭 고정 추적 카메라 |
| 플레이어 | 이동, 상호작용, 짧은 대시 |
| 타워 설치 | 고정 노드 10~12개 |
| 타워 구조 | 역할 3개 (`Attack`, `Control`, `Impact`) |
| 속성 | 3개 (`Fire`, `Water`, `Earth`) |
| 실제 대표 조합 | `Attack + Fire`, `Control + Water`, `Impact + Earth` |
| 타워 단계 | 1~5단계 |
| 적 종류 | 일반, 빠른 적, 탱커, 보스 |
| 웨이브 | 6웨이브 |
| 자원 | 골드, 코어 체력, 보정 게이지 |
| 핵심 기능 | 랜덤 소환, 3합성, 판매, 웨이브 진행 |
| 최소 연출 | 소환 연출, 합성 연출, 5단계 강조 연출, 보스 등장 연출 |
| UI | HUD, 상호작용 프롬프트, 합성 가능 표시, 웨이브 패널 |

### 1.2 왜 여기까지 줄이는가

- 속성 3개와 역할 3개만으로도 전투 감각 차이를 충분히 만들 수 있다.
- 적 타입을 4종으로 줄이면 웨이브 밸런싱과 시각 가독성이 안정된다.
- 플레이어 공격을 MVP에서 제외하면 조작 부담과 전투 밸런스 복잡도가 크게 줄어든다.
- 6웨이브만으로도 `초반 구축 -> 중반 합성 -> 보스 클라이맥스` 구조를 확인할 수 있다.
- 고정 노드 수를 10~12개로 제한하면 보드 관리, 합성 판단, UI 가독성이 훨씬 좋아진다.

### 1.3 이번 MVP에서 제외하는 것

- 플레이어 기본 공격
- 플레이어 액티브 스킬
- 원거리 방해형 적
- 엘리트 별도 분기 적
- 다중 경로 맵
- 자유 배치 타워
- 추가 속성 (`Dark`, `Light`, `Forest`, `Metal`)
- 메타 성장, 해금, 유물, 상점
- 타워 고유 특성 롤
- 멀티플레이
- 복수 보스
- 완전 신규 5단계 모델 다수

### 1.4 그래도 반드시 살려야 하는 것

- 캐릭터가 직접 노드 사이를 이동하며 전장을 관리하는 감각
- 랜덤 소환과 3합성의 핵심 루프
- 불/물/땅의 시각적 구분
- 4단계, 5단계에서 오는 확실한 성장 체감
- 보스전에서 "전장이 흔들리는" 한 장면

### 1.5 개발 진입용 내부 체크포인트

실개발은 바로 전체 MVP를 치지 말고, 먼저 `First Playable Slice`를 만든다.

| 항목 | First Playable Slice |
|---|---|
| 맵 | 1개 |
| 노드 | 6개 |
| 타워 | `Attack + Fire` 1종 |
| 단계 | 1~3단계 |
| 적 | 일반 적 1종 |
| 웨이브 | 3개 |
| 기능 | 이동, 설치, 랜덤 소환, 3합성, 적 이동, 골드 획득 |

이 슬라이스에서 "소환 -> 합성 -> 적 처치 -> 다시 소환"이 재밌으면 그 다음에 Water, Earth, Boss, UI를 추가한다.

### 2.2 폴더 구조 운영 원칙

- `Data`에는 ScriptableObject 정의만 둔다.
- `Runtime`에는 실제 런타임 로직만 둔다.
- `Prefabs`는 기능별로 나누고, 씬 참조가 강한 프리팹은 `World` 아래에 둔다.
- `Presentation`은 연출, VFX 트리거, 카메라 피드백을 담당하고 게임 규칙 로직과 분리한다.
- `Debug`는 개발용 디버그 HUD, 시드 고정, 웨이브 스킵 같은 도구만 둔다.

### 2.3 시작 시점에 꼭 만들 씬

- `Bootstrap`: 공용 초기화, 첫 씬 로딩
- `MVP_Arena`: 실제 플레이용 메인 씬

## 3. 우선 구현해야 할 시스템 순서

실제 구현은 아래 순서를 권장한다. 이유는 `전투 코어 루프`를 가장 빨리 검증하면서도, 각 시스템 의존성을 최소화할 수 있기 때문이다.

1. `프로젝트 부트스트랩`
2. `정적 데이터 정의`
3. `런 상태와 자원 시스템`
4. `플레이어 이동과 상호작용`
5. `타워 노드 보드`
6. `적 이동과 체력`
7. `웨이브 스폰`
8. `타워 공격 런타임`
9. `랜덤 소환`
10. `3합성 및 판매`
11. `속성 효과`
12. `HUD와 월드 UI`
13. `게임 흐름과 보스`
14. `연출/비주얼 패스`

### 3.1 이 순서를 쓰는 이유

- `1~5`까지 끝나면 전장 공간과 상호작용 뼈대가 생긴다.
- `6~9`까지 끝나면 적을 잡는 기본 전투 루프가 성립한다.
- `10~11`에서 게임의 차별점인 합성과 속성 재미가 들어간다.
- `12~14`는 읽힘, 클라이맥스, 보는 맛을 올리는 단계다.

## 4. Task 분해 원칙

Task는 아래 기준으로 쪼갠다.

- 한 Task는 한 가지 책임만 가진다.
- 한 Task는 가급적 `1~4개 스크립트` 안에서 끝난다.
- Task 완료 시 수동 테스트가 바로 가능해야 한다.
- Codex에는 한 번에 한 Task씩 맡긴다.
- Task 완료 후에는 다음 Task 전에 씬에서 직접 검증한다.

## 5. Task 상세 목록

### T01. 프로젝트 부트스트랩

- 목적: Unity 프로젝트의 기본 실행 구조와 폴더, 메인 씬 진입 흐름을 만든다.
- 입력: MVP 씬 구성안, 폴더 구조안, 시작 씬 목록
- 출력: `Bootstrap` 씬, `MVP_Arena` 씬, 기본 실행 설정
- 관련 스크립트:
  - `BootstrapInstaller.cs`
  - `GameSceneLoader.cs`
  - `StartupConfig.cs`
- 완료 조건:
  - 프로젝트를 열면 `Bootstrap`을 통해 `MVP_Arena`로 진입할 수 있다.
  - 폴더 구조가 문서 기준으로 정리되어 있다.
  - 빈 씬이 아니라 플레이 영역의 기본 바닥과 카메라가 존재한다.
- 테스트 방법:
  - Unity에서 Play를 눌러 오류 없이 `MVP_Arena`가 로드되는지 확인
  - Console에 예외가 없는지 확인
  - 씬 전환 후 카메라가 정상 위치에 있는지 확인

### T02. 정적 데이터 레이어

- 목적: 타워, 속성, 적, 웨이브를 하드코딩하지 않고 ScriptableObject로 정의할 수 있게 한다.
- 입력: MVP 잠금 범위의 타워 역할 3개, 속성 3개, 적 4종, 웨이브 6개
- 출력: 데이터 에셋 생성이 가능한 ScriptableObject 타입
- 관련 스크립트:
  - `ElementDefinition.cs`
  - `TowerRoleDefinition.cs`
  - `TowerTierDefinition.cs`
  - `EnemyDefinition.cs`
  - `WaveDefinition.cs`
- 완료 조건:
  - 에디터에서 Fire/Water/Earth 데이터를 만들 수 있다.
  - 타워 단계별 공격력, 사거리, 공속 같은 수치를 데이터로 입력할 수 있다.
  - 웨이브 스폰 정보를 코드 수정 없이 에셋으로 바꿀 수 있다.
- 테스트 방법:
  - 에디터에서 ScriptableObject 생성 메뉴가 정상 노출되는지 확인
  - 각 데이터 에셋을 만들어 Inspector에서 참조가 저장되는지 확인
  - 값 수정 후 Play Mode에서 로딩 가능한지 확인

### T03. 런 상태 및 자원 시스템

- 목적: 현재 판의 골드, 코어 체력, 보정 게이지를 관리하는 런타임 상태를 만든다.
- 입력: 시작 골드, 코어 체력, 소환 비용, 판매 환급 규칙
- 출력: 자원 조회/변경 API와 상태 이벤트
- 관련 스크립트:
  - `RunStateService.cs`
  - `GoldWallet.cs`
  - `CoreHealth.cs`
  - `PityGauge.cs`
- 완료 조건:
  - 골드 증가/감소가 중앙 상태에서 관리된다.
  - 코어 체력이 0이 되면 패배 판정을 낼 수 있는 상태가 된다.
  - 보정 게이지가 증가/소모 규칙에 따라 갱신된다.
- 테스트 방법:
  - 임시 디버그 버튼 또는 인스펙터 호출로 골드 증감 확인
  - 코어 체력 감소 시 0 이하로 안정적으로 clamp 되는지 확인
  - 보정 게이지 증가/초기화 흐름을 로그로 확인

### T04. 플레이어 이동 및 상호작용

- 목적: 플레이어가 전장을 돌아다니며 노드와 타워에 접근할 수 있는 조작 기반을 만든다.
- 입력: 입력 매핑, 이동 속도, 대시 거리, 상호작용 반경
- 출력: 이동 가능한 플레이어 프리팹과 상호작용 타겟 탐색
- 관련 스크립트:
  - `PlayerInputRouter.cs`
  - `PlayerMover.cs`
  - `PlayerDashController.cs`
  - `PlayerInteractionSensor.cs`
- 완료 조건:
  - WASD 이동이 자연스럽다.
  - 플레이어가 근처 노드를 감지할 수 있다.
  - 대시가 과도한 이동 스킵 없이 짧고 명확하게 동작한다.
- 테스트 방법:
  - 키보드 입력으로 1분 이상 이동했을 때 오작동이 없는지 확인
  - 노드 근처 진입/이탈 시 감지 대상이 정상 변경되는지 확인
  - 대시 연속 입력 시 지형 관통이나 튕김이 없는지 확인

### T05. 타워 노드 보드 시스템

- 목적: 맵 위의 고정 노드 상태를 관리하고 설치 가능/점유/합성 가능 여부를 추적한다.
- 입력: 노드 위치 목록, 노드 수, 초기 상태
- 출력: 노드 점유 상태와 설치 대상 조회 API
- 관련 스크립트:
  - `TowerNode.cs`
  - `TowerBoardService.cs`
  - `TowerNodeOccupancy.cs`
  - `TowerNodeView.cs`
- 완료 조건:
  - 맵의 노드가 플레이어 상호작용 대상으로 인식된다.
  - 빈 노드와 점유 노드를 구분할 수 있다.
  - 특정 노드에 어떤 타워가 올라가 있는지 런타임에서 추적된다.
- 테스트 방법:
  - 씬에 10개 이상 노드를 배치하고 상태 전환 로그 확인
  - 타워 설치 전/후 노드 상태가 정확히 바뀌는지 확인
  - 점유 노드 중복 설치가 차단되는지 확인

### T06. 적 이동 및 생존 시스템

- 목적: 적이 경로를 따라 이동하고 피해를 받으며, 죽거나 코어에 도달할 수 있는 기본 적 시스템을 만든다.
- 입력: 경로 포인트, 적 이동 속도, 체력, 코어 피해량, 처치 보상
- 출력: 기본 적 프리팹과 사망/도달 이벤트
- 관련 스크립트:
  - `PathLane.cs`
  - `EnemyMover.cs`
  - `EnemyHealth.cs`
  - `EnemyRewardOnDeath.cs`
- 완료 조건:
  - 적이 경로를 따라 끝까지 이동한다.
  - 체력이 0이 되면 사망 이벤트와 골드 보상이 발생한다.
  - 코어 도달 시 코어 체력이 감소한다.
- 테스트 방법:
  - 적 10마리를 스폰해 경로 이탈이 없는지 확인
  - 임시 데미지 입력으로 적 사망 처리 확인
  - 코어 도달 시 자원 시스템과 연결되는지 확인

### T07. 웨이브 스폰 시스템

- 목적: WaveDefinition을 읽어 적을 정해진 순서와 간격으로 스폰한다.
- 입력: 웨이브 데이터, 스폰 위치, 웨이브 시작 규칙
- 출력: 웨이브 진행 상태, 남은 적 수, 웨이브 종료 이벤트
- 관련 스크립트:
  - `WaveController.cs`
  - `EnemySpawner.cs`
  - `WaveRuntimeState.cs`
- 완료 조건:
  - 6개 웨이브를 순차적으로 실행할 수 있다.
  - 웨이브 종료와 다음 웨이브 시작을 구분할 수 있다.
  - 현재 남은 적 수와 웨이브 인덱스를 조회할 수 있다.
- 테스트 방법:
  - 3개 테스트 웨이브부터 실행해 스폰 간격 확인
  - 마지막 적 처치 후 웨이브 종료 이벤트가 정확한지 확인
  - 여러 웨이브를 연속 재생해 누락 스폰이 없는지 확인

### T08. 타워 공격 런타임

- 목적: 타워가 적을 탐지하고 공격하며 피해를 주는 기본 전투 루프를 만든다.
- 입력: 타워 사거리, 공속, 공격력, 투사체 속도 또는 즉발 규칙
- 출력: 타워의 타겟팅, 발사, 명중, 피해 적용
- 관련 스크립트:
  - `TowerRuntime.cs`
  - `TowerTargetSelector.cs`
  - `TowerAttackController.cs`
  - `ProjectileMover.cs`
- 완료 조건:
  - 설치된 타워가 사거리 안 적을 찾아 공격한다.
  - 적이 맞으면 체력이 감소한다.
  - 여러 타워가 동시에 동작해도 타겟팅이 깨지지 않는다.
- 테스트 방법:
  - 1개 타워 대 1개 적에서 기본 공격 루프 확인
  - 3개 타워 대 10개 적 상황에서 오작동 여부 확인
  - 적이 사거리에서 나가거나 죽었을 때 타겟이 정상 해제되는지 확인

### T09. 랜덤 소환 시스템

- 목적: 플레이어가 빈 노드에서 골드를 써서 타워를 랜덤 생성할 수 있게 한다.
- 입력: 현재 골드, 소환 비용, 활성 조합 풀, 대상 노드
- 출력: 생성된 타워 인스턴스와 소환 결과
- 관련 스크립트:
  - `SummonService.cs`
  - `SummonPool.cs`
  - `SummonRequestHandler.cs`
  - `TowerFactory.cs`
- 완료 조건:
  - 플레이어가 빈 노드에서 소환을 실행할 수 있다.
  - 골드가 부족하면 소환이 차단된다.
  - 현재 MVP 대표 조합 3개 중 하나가 랜덤으로 생성된다.
- 테스트 방법:
  - 빈 노드에서 20회 이상 연속 소환해 정상 생성 여부 확인
  - 골드 부족 상태에서 소환 불가 처리 확인
  - 점유 노드에서 소환이 차단되는지 확인

### T10. 3합성 및 판매 시스템

- 목적: 같은 조합, 같은 단계 타워 3개를 상위 단계 1개로 합성하고, 필요 시 판매할 수 있게 한다.
- 입력: 보드 위 타워 상태, 합성 규칙, 판매 환급 비율
- 출력: 상위 단계 타워 생성, 기존 타워 제거, 판매 골드 반환
- 관련 스크립트:
  - `TowerMergeQuery.cs`
  - `TowerMergeService.cs`
  - `TowerSellService.cs`
  - `MergeCandidateMarker.cs`
- 완료 조건:
  - 같은 조합, 같은 단계 타워 3개를 찾을 수 있다.
  - 플레이어 입력으로 합성이 실행된다.
  - 판매 시 노드가 비워지고 골드가 환급된다.
- 테스트 방법:
  - 같은 타워 3개를 배치해 1->2, 2->3 단계 합성 확인
  - 단계나 조합이 다르면 합성되지 않는지 확인
  - 판매 후 다시 소환 가능한 빈 노드로 돌아가는지 확인

### T11. 속성 효과 시스템

- 목적: 역할 코어 위에 Fire, Water, Earth 속성 효과를 얹어 대표 조합의 차이를 만든다.
- 입력: 공격 이벤트, 속성 정의 데이터, 상태이상 강도
- 출력: 화상, 둔화, 넉백 또는 충격파 효과 적용
- 관련 스크립트:
  - `ElementEffectProcessor.cs`
  - `FireEffectHandler.cs`
  - `WaterEffectHandler.cs`
  - `EarthEffectHandler.cs`
- 완료 조건:
  - Fire는 도트 또는 폭발 보조 효과를 적용한다.
  - Water는 이동 둔화를 적용한다.
  - Earth는 넉백 또는 짧은 충격파 반응을 적용한다.
  - 속성별로 맞았을 때 다른 피드백을 느낄 수 있다.
- 테스트 방법:
  - 같은 스탯 타워에 속성만 바꿔 적 반응 차이 비교
  - 둔화 중첩과 넉백 적용이 이동 시스템을 망가뜨리지 않는지 확인
  - Fire 도트가 사망 처리와 충돌하지 않는지 확인

### T12. HUD 및 월드 UI

- 목적: 플레이어가 현재 판 상태를 읽고 소환, 합성, 웨이브 정보를 이해할 수 있는 최소 UI를 제공한다.
- 입력: 런 상태 이벤트, 웨이브 상태, 노드 상태, 합성 가능 상태
- 출력: HUD, 상호작용 프롬프트, 합성 가능 표시, 웨이브 패널
- 관련 스크립트:
  - `HudPresenter.cs`
  - `InteractPromptPresenter.cs`
  - `WavePanelPresenter.cs`
  - `TowerWorldIndicator.cs`
- 완료 조건:
  - 골드, 코어 체력, 웨이브 정보가 화면에 표시된다.
  - 플레이어가 노드 근처에서 할 수 있는 행동을 알 수 있다.
  - 합성 가능한 타워가 시각적으로 표시된다.
- 테스트 방법:
  - 플레이 도중 자원 수치가 즉시 갱신되는지 확인
  - 합성 가능/불가 상태 전환 시 표시가 정확한지 확인
  - HUD가 카메라와 해상도 변화에 크게 깨지지 않는지 확인

### T13. 보정 소환 시스템

- 목적: 초반 운빨 망을 줄이기 위해 보정 게이지와 가이드 소환 로직을 붙인다.
- 입력: 현재 보드 상태, 최근 소환 기록, 보정 게이지 값
- 출력: 가중치 조정된 소환 결과 또는 2선택 소환 상태
- 관련 스크립트:
  - `SummonBalanceDirector.cs`
  - `SummonHistoryTracker.cs`
  - `GuidedSummonOptionBuilder.cs`
- 완료 조건:
  - 초반 4회 내 최소 한 번은 합성 가능성이 생기도록 보정된다.
  - 장시간 합성 후보가 없으면 보정 게이지가 의미 있게 작동한다.
  - 완전 고정 결과가 아니라 "도와주는 랜덤"으로 남는다.
- 테스트 방법:
  - 고정 시드로 반복 실행해 초반 소환 분포 확인
  - 의도적으로 불리한 보드를 만든 뒤 보정 발동 확인
  - 보정 전후 체감 차이를 플레이 테스트로 비교

### T14. 게임 흐름 및 보스전

- 목적: 승리/패배 조건, 마지막 보스전, 노드 비활성화 기믹을 포함한 런의 클라이맥스를 만든다.
- 입력: 웨이브 종료 상태, 코어 체력, 보스 데이터, 노드 비활성화 규칙
- 출력: 보스 등장, 승패 판정, 결과 화면 전환
- 관련 스크립트:
  - `RunFlowController.cs`
  - `BossController.cs`
  - `BossPhaseController.cs`
  - `NodeDisableEffect.cs`
- 완료 조건:
  - 마지막 웨이브에서 보스가 등장한다.
  - 보스가 일정 주기로 노드를 비활성화한다.
  - 코어 파괴 또는 보스 처치 시 런이 종료된다.
- 테스트 방법:
  - 일반 웨이브 종료 후 보스 진입이 정확한지 확인
  - 노드 비활성화가 타워 상태와 충돌하지 않는지 확인
  - 승리/패배 양쪽 결과 화면이 정상 동작하는지 확인

### T15. 연출 및 비주얼 패스

- 목적: 시스템은 이미 동작하는 상태에서 소환, 합성, 단계 상승, 보스 등장 연출을 넣어 보는 맛을 만든다.
- 입력: 타워 단계 상태, 소환 이벤트, 합성 이벤트, 보스 등장 이벤트
- 출력: 최소 VFX, 머티리얼 변화, 카메라 피드백, 화면 강조
- 관련 스크립트:
  - `TowerTierVisualController.cs`
  - `SummonVfxPlayer.cs`
  - `MergeVfxPlayer.cs`
  - `BossIntroSequence.cs`
- 완료 조건:
  - 소환 시 노드 점등과 짧은 생성 연출이 있다.
  - 합성 시 시각적으로 "승격"이 느껴진다.
  - 5단계 타워는 최소한의 전용 강조 연출이 있다.
  - 보스 등장 시 카메라 또는 라이트 변화가 있다.
- 테스트 방법:
  - 합성 10회 이상 반복 시 연출 누락이나 중복이 없는지 확인
  - 5단계 생성 시 프레임 급락이 없는지 확인
  - 보스 등장 컷이 플레이 흐름을 과하게 끊지 않는지 확인

## 6. 구현 순서별 마일스톤

### M1. First Playable

- 포함 Task: `T01 ~ T09`
- 목표: 1개 타워, 1개 적, 3웨이브로 소환과 처치 루프 성립
- 통과 조건:
  - 빈 노드에 타워를 설치할 수 있다.
  - 적이 경로를 따라 오고 타워가 적을 죽인다.
  - 적 처치 골드로 다시 소환할 수 있다.

### M2. Core MVP

- 포함 Task: `T10 ~ T12`
- 목표: 합성, 속성 차이, 기본 UI까지 붙어 MVP의 핵심 재미가 드러남
- 통과 조건:
  - 3합성이 반복적으로 가능하다.
  - Fire/Water/Earth 전투 감각이 구분된다.
  - HUD만 보고 현재 상황을 판단할 수 있다.

### M3. Fairness & Finish

- 포함 Task: `T13 ~ T15`
- 목표: 보정 소환, 보스전, 연출까지 붙여 한 판 완결 구조 완성
- 통과 조건:
  - 억울한 초반 런이 크게 줄어든다.
  - 보스전이 런의 분명한 클라이맥스로 작동한다.
  - 합성과 5단계가 시각적으로 기억에 남는다.

## 8. 최종 작업 기준

실제 구현 착수 시에는 아래 원칙을 유지한다.

- 한 번에 하나의 Task만 구현한다.
- Task를 시작하기 전에 관련 파일과 기존 구조를 먼저 확인한다.
- 수치는 ScriptableObject나 설정 데이터로 뺀다.
- 기능이 붙을수록 `Core`와 `Presentation`의 경계를 더 엄격하게 지킨다.
- 보드가 막히는 상황, 합성이 안 되는 상황, UI가 읽히지 않는 상황을 우선적으로 줄인다.

이 문서 기준으로 Codex에게 작업을 맡기면, 큰 구조를 무너뜨리지 않고도 순차적으로 MVP를 쌓아갈 수 있다.
